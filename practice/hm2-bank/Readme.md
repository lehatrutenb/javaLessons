Из идей SOLID:
1. Разбиение интерфейсов на меньшие - некоторые классы имлементирует сразу несколько, чтобы не порождать больших
2. D - модули верхних уровней зависят от интерфейсов, там где предполагается вариативноть
3. O - паттерн visitor

Из идей GRASP:
1. Indirection:
   1. объекты надстройки с суффиксом Report - хочется максимально прижать к самим доменом информацию о хранении доменов
        тк очень частно мы захотим обращаться к их множеству - хочется сделать его доступным к изменению
   2. большие классы не общаются напрямую , а зависят от интерфейсов
2. Pure Fabrication - dto объекты в каком-то плане

Из GoF реализованны:
1. Фабрика - создание различных доменных объектов 
2. Шаблонный метод - много шаблонизации внутри экспортеров данных чтобы не писать код дважды
3. Visitor (посетитель) - для csv не было удобной либы и решил через visitor реализовать экспорт объекта
4. Memento - хотим внутри логики хранить BankAccount и Category внутри Operation - но при сохранении меняем облегчаем до memento
5. Facade - хотим объеденить много функций и классов под 1 общий вход
